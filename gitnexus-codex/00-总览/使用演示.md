# 使用演示

## 基础命令

```bash
npx gitnexus analyze
npx gitnexus status
npx gitnexus query "auth validation flow"
npx gitnexus context validateUser
npx gitnexus impact validateUser --direction upstream
npx gitnexus serve
```

## Demo 场景：修改 validateUser

传统 Agent 可能会这样做：

1. grep validateUser。
2. 打开几个文件。
3. 猜调用方和影响面。
4. 直接改。

GitNexus Agent 应该这样做：

1. `query`：先找认证或校验相关流程。
2. `context`：看 validateUser 的定义、调用方、被调用方、参与的流程。
3. `impact`：修改前看上游影响面和风险等级。
4. 修改代码。
5. `detect_changes`：提交前确认影响范围符合预期。

## 演示时的讲法

这段 Demo 的重点不是命令本身，而是工作流变化：Agent 从“凭文本片段猜”变成“先拿结构化证据再改”。
