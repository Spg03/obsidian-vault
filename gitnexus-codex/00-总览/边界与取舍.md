# 边界与取舍

技术分享里一定要主动讲边界。程序员听众会自然追问：静态分析准吗？动态调用怎么办？索引过期怎么办？Embedding 会不会胡召回？Group 匹配会不会误连？

## GitNexus 的强项

GitNexus 擅长捕获大量静态、可预计算、可验证的工程关系：

- 文件结构。
- 符号定义。
- imports。
- 调用链。
- 继承和实现。
- 路由和工具入口。
- 执行流。
- 功能社区。
- API consumer/provider 契约。

这些关系非常适合 Agent 编程，因为 Agent 修改代码前最需要知道的就是“谁依赖谁”和“改了影响谁”。

## 静态分析的边界

静态分析天然会遇到这些问题：

- 反射调用。
- 动态 import。
- 字符串拼接路由。
- 运行时 monkey patch。
- 依赖注入容器里的隐式绑定。
- 框架魔法。
- 多态分发和泛型推断不完整。

所以正确措辞是：

> 静态分析可捕获大量确定性关系，并为不确定关系保留置信度或边界提示。

不要说：

> GitNexus 可以 100% 理解所有调用。

## 索引新鲜度

GitNexus 的图谱来自上一次 analyze。代码变了，索引可能 stale。

这会影响：

- query 结果。
- context 里的调用方。
- impact 的影响范围。
- route_map / shape_check。

所以 Agent 文件里会提醒：如果 GitNexus 工具提示 index stale，应先运行 `npx gitnexus analyze`。

## FTS 和 Embedding 的边界

FTS 和 Embedding 是召回，不是证明。

- BM25 证明的是关键词相关。
- Embedding 证明的是向量相似。
- CALLS 边才表示静态分析认为存在调用关系。

分享时要明确：GitNexus 不是“靠向量搜索猜调用链”。调用链和影响面主要来自图谱关系。

## MCP 约束的边界

AGENTS.md、Skill、tool description 能显著影响 Agent 行为，但不能像权限系统一样强制执行。Agent 仍可能：

- 忘记读 skill。
- 跳过 impact。
- 忽略 HIGH 风险。
- 错解工具输出。

因此仍需要：

- 检查 detect_changes。
- 跑测试和类型检查。
- 人工 review。
- 对高风险修改停下来确认。

## Group 的边界

跨仓库关系通常不是 AST 能直接看出来的，而是通过契约连接：

- HTTP route。
- gRPC service。
- Thrift service。
- Topic / Queue。
- include。
- workspace dependency。
- manifest link。

自动匹配可能漏，也可能误连。显式 manifest link 是重要补充。复杂组织里最好把 Group 视为“跨仓库影响分析辅助系统”，而不是绝对真相源。

## 分享时的可信表达

建议说：

- 默认本地优先。
- 静态分析捕获大量确定性关系。
- 图谱查询减少 Agent 盲改概率。
- 影响分析帮助定位测试和 review 重点。
- 动态行为仍需要测试和运行时验证。

避免说：

- 绝对隐私。
- 100% 精准。
- 自动理解全部业务。
- 不需要测试。
- 比人工 review 更可靠。
